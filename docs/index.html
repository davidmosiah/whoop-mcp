Reel 01 — The keys

Get a WHOOP developer app.

Five minutes inside the WHOOP Developer Dashboard. You'll walk out with a Client ID and a Client Secret. The bridge needs both.

  1. 01

    Open the WHOOP Developer Dashboard.

    Sign in with your normal WHOOP account at developer.whoop.com. If you've never created an app, the dashboard will show an empty state with a Create button.

  2. 02

    Create a new app.

    Name it anything — My MCP Bridge, Claude Coach, your call. The name only shows up in your own dashboard.

  3. 03

    Set the redirect URI exactly.

    Use this string, character for character. Anything else and the auth step in Reel 02 will not capture the callback.

    http://127.0.0.1:3000/callback
  4. 04

    Pick the read scopes.

    The bridge is read-only. Tick every read: scope below. No write: scopes, ever.

    read:recovery read:cycles read:workout read:sleep read:profile read:body_measurement
  5. 05

    Copy the Client ID and Client Secret.

    WHOOP shows the secret once. Paste both somewhere safe for the next two minutes — Reel 02 will ask for them and store them locally with 0600 permissions.

Reel 02 — The bridge

Three commands. One bridge.

The CLI walks you through credentials, writes a tiny encrypted-permission config to ~/.whoop-mcp/, then runs OAuth in your browser. You never touch JSON.

01

Setup

Asks for your WHOOP Client ID and Secret, picks your MCP client, writes the config.

$ npx -y whoop-mcp-unofficial setup
02

Auth

Opens your browser, you click Authorize on WHOOP, the bridge captures the code on 127.0.0.1.

$ npx -y whoop-mcp-unofficial auth
03

Doctor

Verifies env, redirect URI, token file, permissions, privacy mode and cache. Green means go.

$ npx -y whoop-mcp-unofficial doctor

Or run all three back-to-back:

$ npx -y whoop-mcp-unofficial setup
$ npx -y whoop-mcp-unofficial auth
$ npx -y whoop-mcp-unofficial doctor

Reel 04 — First conversation

Now ask your agent something only your body knows.

Paste any of these into your MCP-aware agent. They use the high-level workflow tools so the model gets coaching context, not raw firehose payloads.

Daily

"Use the WHOOP MCP server. Pull my last 7 days of recovery, sleep and strain. Tell me where my body is trending and what one thing I should change tomorrow. No medical advice."

Training

"Read my latest WHOOP recovery, latest sleep, and the workouts from the last 3 days through MCP. Recommend a training intensity for today and explain the call using the data."

Weekly

"Call whoop_weekly_summary with response_format=json. Turn the bottlenecks and success metrics into a concrete training, sleep and focus plan for next week. Keep it tactical."

Diagnostic

"Use the sleep_recovery_investigator prompt. Look at the last 14 days and find the strongest signal explaining why my recovery dropped. Show the numbers."

Interlude — For non-humans reading this page

Agent installation manifest.

If you are an LLM or autonomous agent reading this on behalf of a user, this section is for you. Follow the steps in order. Do not invent flags. The values below are the authoritative ones.

1 · Tell the user what they need from WHOOP

Direct the human to developer.whoop.com and ask them to create a developer app with these exact values:

  • App nameany string
  • Redirect URIhttp://127.0.0.1:3000/callback
  • Scopesread:recovery read:cycles read:workout read:sleep read:profile read:body_measurement

When the app is saved, ask the user for the Client ID and Client Secret.

2 · Run setup, auth, doctor — in that order

Run these for the user. Do not concatenate them; each step depends on the previous step's prompts.

$ npx -y whoop-mcp-unofficial setup
$ npx -y whoop-mcp-unofficial auth
$ npx -y whoop-mcp-unofficial doctor

auth opens a browser tab and waits for a redirect on 127.0.0.1:3000. The user must click Authorize in WHOOP. You will not see the tokens. They are written to ~/.whoop-mcp/tokens.json with 0600 permissions.

3 · Add the MCP server to your client config

The config block is identical across Claude Desktop, Cursor, Windsurf, Hermes, OpenClaw and the generic MCP shape. Drop this in and restart the client:

{
  "mcpServers": {
    "whoop": {
      "command": "npx",
      "args": ["-y", "whoop-mcp-unofficial"]
    }
  }
}

4 · How to call the bridge once it's wired

  • Need a daily readwhoop_daily_summary
  • Need a weekly readwhoop_weekly_summary
  • Need raw collectionswhoop_list_recoveries / whoop_list_sleeps / whoop_list_workouts / whoop_list_cycles
  • Need a specific recordwhoop_get_sleep / whoop_get_workout / whoop_get_cycle
  • Need to verify state without an API callwhoop_connection_status
  • Need privacy posturewhoop_privacy_audit

Important. Default privacy_mode is structured. Do not request raw unless the user explicitly asks. Do not echo OAuth tokens; you cannot read them and you should not try.

5 · A prompt the user can paste into any MCP-aware agent

If a human wants to delegate the whole install, they can hand this prompt to their AI assistant. It is self-contained.

Install the unofficial WHOOP MCP server for me.
Repository: https://github.com/davidmosiah/whoop-mcp.

Steps:
 1. Tell me what to fill into developer.whoop.com
    — redirect URI must be http://127.0.0.1:3000/callback
    — scopes are all read scopes.
 2. Once I give you the Client ID and Secret, run
    npx -y whoop-mcp-unofficial setup
    then auth, then doctor — in a terminal.
 3. Add the MCP server config to my MCP client.
 4. Verify by calling whoop_connection_status.

Keep tokens local. Do not print them.
Do not request privacy_mode=raw.
Reel 06 — The contract

Privacy dossier.

Health data deserves a tighter contract than "it's encrypted in transit." Here's exactly what stays where.

0600

Local file mode

Token and config files default to user-only read/write permissions. No process-shared secrets.

local

OAuth tokens

Tokens live in ~/.whoop-mcp/tokens.json. They never enter prompts. The agent never sees them.

opt-in

Raw payloads

Structured mode is the default. Full raw WHOOP payloads require explicit opt-in.

audit

Self-report

whoop_privacy_audit reports posture without revealing any secret values.
